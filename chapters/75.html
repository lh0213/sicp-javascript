4.1.2. Representing Statements and Expressions

The evaluator is reminiscent of the symbolic differentiation program discussed in Section . Both programs operate on symbolic expressions. In both programs, the result of operating on a compound expression is determined by operating recursively on the pieces of the expression and combining the results in a way that depends on the type of the expression. In both programs we used data abstraction to decouple the general rules of operation from the details of how expressions are represented. In the differentiation program this meant that the same differentiation function could deal with algebraic expressions in prefix form, in infix form, or in some other form. For the evaluator, this means that the syntax of the language being evaluated is determined solely by the functions that classify and extract pieces of expressions.

Here is the specification of the syntax of our language:

  • The self-evaluating items are numbers, strings, boolean values, and the special value []. 
    function is_self_evaluating(stmt) {
   return stmt === [] ||
    typeof stmt === "number" ||
    typeof stmt === "string" ||
    typeof stmt === "boolean";
}
    [Open in IDE]
  • The function variable tests whether the given statement is a variable expression, and the function variable_name accesses the name of the variable as a JavaScript string. 
    function is_variable(stmt) {
   return is_tagged_object(stmt,"variable");
}
function variable_name(stmt) {
   return stmt.name;
}
    [Open in IDE]
    The function variable is defined in terms of the function is_tagged_object, which identifies objects that have a designated string under the property tag: 
    function is_tagged_object(stmt,the_tag) {
   return typeof stmt === "object" &&
   stmt.tag === the_tag;
}
    [Open in IDE]
    Thus, a tagged object tagged as variable looks like this: { tag: "variable", ... } Note that the test typeof stmt === "object" is necessary, because undefined . tag generates the exception undefined has no properties.[1]
  • Assignments have the form var = value: 
    function is_assignment(stmt) {
   return is_tagged_object(stmt,"assignment");
}
function assignment_variable(stmt) {
   return stmt.variable;
}
function assignment_value(stmt) {
   return stmt.value;
}
    [Open in IDE]
  • Definitions have the form var ^variable^ = ^value^; or the form function ^variable^(^parameter$_{1}$^, ^$\ldots$^, ^parameter$_{n}$^) { ^body^ } Here, we treat the latter form (standard function definition) as syntactic sugar[2] for var ^variable^ = function(^parameter$_{1}$^, ^$\ldots$^, ^parameter$_{n}$^) { ^body^ }; The corresponding syntax functions are the following: 
    function is_var_definition(stmt) {
   return is_tagged_object(stmt,"var_definition");
}
function var_definition_variable(stmt) {
   return stmt.variable;
}
function var_definition_value(stmt) {
   return stmt.value;
}
    [Open in IDE]
  • Function definitions are objects tagged with the string function_definition: 
    function is_function_definition(stmt) {
   return is_tagged_object(stmt,"function_definition");
}
function function_definition_parameters(stmt) {
   return stmt.parameters;
}
function function_definition_body(stmt) {
   return stmt.body;
}
    [Open in IDE]
  • return statements are objects tagged with the string "return_statement": 
    function is_return_statement(stmt) {
   return is_tagged_object(stmt,"return_statement");
}
function return_statement_expression(stmt) {
   return stmt.expression;
}
    [Open in IDE]
  • Conditional statements begin with if and have a predicate, a consequent, and an (optional) alternative. If the expression has no alternative part, we provide undefined as the alternative.[3] 
    function is_if_statement(stmt) {
   return is_tagged_object(stmt,"if");
}
function if_predicate(stmt) {
   return stmt.predicate;
}
function if_consequent(stmt) {
   return stmt.consequent;
}
function if_alternative(stmt) {
   return stmt.alternative;
}
    [Open in IDE]
  • A sequence is a list of statements. 
    function is_sequence(stmt) {
   return is_list(stmt);
}
function is_last_statement(stmts) {
   return is_empty_list(tail(stmts));
}
function first_statement(stmts) {
   return head(stmts);
}
function rest_statements(stmts) {
   return tail(stmts);
}
    [Open in IDE]
  • A function application is an object tagged with the string "application". We provide access functions for the operator, the operands, and three functions for iterating through the operand list: 
    function is_application(stmt) {
   return is_tagged_object(stmt,"application");
}
function operator(stmt) {
   return stmt.operator;
}
function operands(stmt) {
   return stmt.operands;
}
function no_operands(ops) {
   return is_empty_list(ops);
}
function first_operand(ops) {
   return head(ops);
}
function rest_operands(ops) {
   return tail(ops);
}
    [Open in IDE]

Rewrite eval so that the dispatch is done in data-directed style. Compare this with the data-directed differentiation function of Exercise . (You may use the head of a compound expression as the type of the expression, as is appropriate for the syntax implemented in this section.) Recall the definitions of the special forms and and or from Chapter 1:
  • $\langle e_1 \rangle$ && $\langle e_2 \rangle$: The expression $\langle e_1 \rangle$ is evaluated first. If it evaluates to false, false is returned; the expression $\langle e_2 \rangle$ is not evaluated. If it evaluates to true, the value of $\langle e_2 \rangle$ is returned.
  • $\langle e_1 \rangle$ || $\langle e_2 \rangle$: The expression $\langle e_1 \rangle$ is evaluated first. If it evaluates to true, true is returned; the expression $\langle e_2 \rangle$ is not evaluated. If it evaluates to false, the value of $\langle e_2 \rangle$ is returned.
Install && and || as new special forms for the evaluator by defining appropriate syntax functions and evaluation functions eval_and and eval_or Alternatively, show how to implement && and || as derived expressions.
[1] The primitive value [] behaves similarly, whereas stmt.tag properly returns false for numbers and boolean values as stmt.
[2] In actual JavaScript, there is a subtle difference between the two forms. The interpretation of function definition statements involves reordering of sequence statements, which I feel is too complicated to be handled here.
[3] We will support the use of the variable undefined in statements to be evaluated by binding them in the global environment; see Section . Note that [], true, and false are keywords and handles as self-evaluating expressions.
